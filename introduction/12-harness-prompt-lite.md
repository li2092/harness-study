# Agent Harness 通用 TDD 提示词

三段通用指令,输给编码 AI 可以直接按此搭建和落地一套 agent runtime。
三段之间是上下游关系:先搭通用 runtime,再针对场景收窄,最后按成本结构分配资源。

---

## 第一段:通用 Agent Runtime(不绑语言)

```
搭一个 agent runtime,Python 最快两天跑通、TypeScript 方便接前端四天、
Rust 稳但工程量大要一周。模块划三大块加两块支撑:engine 跑循环、llm
做多供应商抽象、tools 管注册和执行,storage 存 session 和 checkpoint,
types 集中定义消息和工具签名。

engine 的主循环就一个 ReAct:发消息→收回复→有工具调用就执行→结果回
写→判终止。再加一个状态机管多轮:等用户输入、等用户确认、跑完、出错。
每轮开头先查取消信号,stop 要即时生效。

llm 抽成一个统一接口,各家差异在适配层处理。thinking 参数每家写法不
同,用 extra 字段各自配。assistant 消息协议不能乱——tool_calls 后面
必须紧跟 tool_result 中间别插别的,否则下一轮请求直接 400。国内 API
一律显式绕代理,单独设 no_proxy 挡不住小写环境变量。

上下文两层压缩。消息多了先清旧工具结果(micro),token 估算超窗口 60-
70% 触发小模型摘要(auto)。压完必须把任务目标和 system 身份重注入,
不然几十轮后 agent 会偏题,以为自己在干别的事。

工具层每步重新加载列表别缓存,POST hook 里注册的新工具下一轮要看得
见。单工具输出分别设硬上限(读文件 20k、搜索 10k、shell 15k 起步),
超限送外置存储返 ID。shell 和 git 命令一律走 execFile 加数组参数,
字符串拼接拒绝,防注入。

可观测性从第一行开始埋。埋在决策点不在执行点——机制判断"该不该做"
的地方。四种状态:触发执行了、条件不满足没执行、触发但拦截了、执行
出错。没事件是最危险的信号,说明代码路径根本没走到。

循环检测拦重复工具调用,介入消息先攒到队列里等 tool_result 处理完统
一推,别中途破坏消息协议。

并发别在锁里同步等长任务。锁里只 take 出 engine 立刻释放,异步跑并
监听取消信号,成功、取消、出错三条路径都保证 engine 归还。

前端接入时后端是唯一真相源,别做前端状态快照。切换 session 从后端
拉 last_messages 和 is_running。

单 agent 跑通按上面三种语言估时,加上观测埋点、循环检测、多 agent 隔
离再加一周。接强模型,弱模型带不动多轮工具调用。
```

## 第二段:从通用到场景的收窄(不绑行业)

```
通用 runtime 跑通后,针对具体业务场景做收窄。不管是客服、标书、质检、
运维、翻译还是数据分析,方法一样,八步做完。

第一步建评测集。3-5 个典型任务起步,每个任务有可自动化的通过/失败判
定,有预期工具调用序列和最终数据库状态。评测集和代码一起演化,不要
等"做完再测"。

第二步工具收窄。通用 runtime 可能挂 40-50 个工具,业务场景一般只用
8-15 个。其他工具显式禁用写进 disabledTools 白名单,不是放着不调
——留在上下文里会分散模型注意力,选错工具的概率上升。

第三步工具描述强化。每个业务工具的 description 里直接列必填字段名、
类型、枚举值。字段是 user_id 还是 userId、是数组还是对象、枚举有几
个值,别让模型猜。猜错一次就浪费一轮,这是 ROI 最高的单点优化,比改
system prompt 有效一个数量级。

第四步策略规则引擎。业务规则不要全堆在 system prompt 里,模型跑几十
轮就忘。做成"调用 X 工具前注入结构化提醒"——取消操作前弹资格条件
检查,删除前弹二次确认,赔偿前弹前置条件。即时提醒比全局声明有效得
多。

第五步提示词分层。系统层写角色和能力边界,场景层写业务规则和禁用行
为,执行层写输出格式(比如必须报数字、必须同语言回复、必须等用户确
认)。三层分开,方便单独调整和 A/B。

第六步开推理模式(如果模型支持)。对多步判断、数值计算、规则检查密
集的场景,推理模式能直接拉十个百分点。简单问答场景不用开,浪费 token。

第七步跑消融实验。每加一个机制,关掉再跑一次评测集,看分数掉多少。
正贡献保留,负贡献砍掉,接近零的看维护成本决定。关键在这一步——有
些"看起来正确"的机制,关掉反而提升,单元测试根本看不出来。

第八步失败归因分维度。不止看通过率,还要看:工具调用参数对不对、数
据库最终状态对不对、回复里该说的数字/条款有没有说出来、该走的流程
走完没有。单维度评测会掩盖真问题。数据库对了但没说出数字,在客户看
来就是没办成事。

这套收窄做完,通用 runtime 在特定场景下能多拿 10-20 个百分点。比换
模型便宜,比重写框架快。
```

## 第三段:成本结构(什么先砸、什么后砸)

```
资源不要平摊到每个机制。瓶颈在哪里就砸哪里。顺序错了,花同样的钱拿
不到同样的结果。

先砸评测集。没评测等于闭眼调优,后面所有改动都说不清"到底有没有用"。
评测集粗糙到只看通过率也比没有强。三五个能自动判定的典型任务,跑起
来就行。每次改动跑一遍,慢慢补充边界 case。

然后砸工具描述。这是单点 ROI 最高的位置——模型猜字段名猜错一次就浪
费一轮,几十轮对话下来浪费可能是一半轮数。写清必填字段、类型、枚举
值,比改十次 system prompt 有效。

接着砸策略规则引擎。业务规则做成"调用前注入"。几十轮之后模型会忘
system prompt 里的规则,即时注入的不会忘。这一步做完之后再看模型表
现,比直接换模型便宜得多。

推理模式开不开,看任务是不是多步判断。规则检查、数值计算、多条件
判断密集的场景开,能拉十个百分点。简单问答场景别开,贵、慢、没收益。

模型升级放到最后。从便宜模型开始,调到瓶颈再换。模型贵、不可逆(换
了重新跑评测)、对大部分场景提升有限——真正卡通过率的一般不是模型
能力,是外围机制没调好。大部分团队的误区是第一步就换模型,结果钱花
了分数没涨。

基础设施(压缩、观测、循环检测、锁和取消)是入场券不是投入。不做一
切都谈不上。但做到"能跑就行"的程度够用,过度工程反而拖慢迭代。

按场景类型看分配:
  客服和规则执行类     先砸工具描述 + 策略规则,再考虑推理模式
  代码和复杂规划类     先砸推理模式 + 强模型,工具描述和规则次之
  长对话和多轮类       先砸压缩 + 观测,不然跑几十轮自己就崩
  高并发和低延迟类     选便宜模型 + 工具输出缓存 + 简化工具集

最后一点最反直觉:负贡献机制要及时砍。有些机制做了反而掉分,继续维
护等于双重损失——占资源还拖通过率。消融一测就看出来。舍不得砍"看
起来正确的代码"是常见错误。
```

## 三段的使用顺序

1. 先用第一段搭通用 runtime。这一步不考虑具体业务,只把框架跑起来。
2. 通用 runtime 能对着任意工具集发起 ReAct 循环之后,用第二段做场景
   收窄。这一步和业务一起演化,不要等"通用版做到完美"再收窄。
3. 第三段在任何阶段都可以对照,它决定的不是"做什么",是"做哪个先"
   和"花哪份钱"。顺序错了,通用 runtime 搭得再好,业务上也出不了数。

三段合起来的总逻辑:**先搭一个能跑的东西,再针对场景削出锋利的刀,
然后把钱花在刀刃上。**

## 2026-04-26 补充：机制准入纪律

结合 PCS Sanity Checks 和 Plan-RewardBench 的思路，Agent-Z 后续不能只凭“机制看起来合理”就默认启用。每个机制进入主 harness 默认路径前，至少要满足四条：

1. **能改变真实行为**：不是只增加 DTO、schema 或日志字段。
2. **有 on/off ablation**：关掉后能看到 pass_rate、cost、latency、reliability 的变化。
3. **有机器可读 trace**：记录触发原因、规则 ID、输入摘要和结果。
4. **过稳定性检查**：N 次重跑、扰动 prompt、打乱无关字段后，收益不能只靠偶然通过。

达不到四条的机制可以保留为实验开关，但不进入默认 profile。负贡献或不稳定贡献要及时撤回，不能让主 harness 被“半接通机制”拖慢。
