# Harness Prompt · Agent Harness 落地 Spec（给 agent 的可执行版）

这份文件是正文《Harness Study》的**可执行配套件**。正文讲"有哪些件、为什么这样设计"，这份讲"按什么顺序动手、每步怎么验证做对了"。

**怎么用**：把这份文件连同你的具体场景需求（"帮我做一个 X 场景的 agent"）一起交给编码 agent。agent 按 Phase 0 → 1 → 2 → 3 顺序执行，每个 Phase 末尾的**完成判据（gate）不过，不进下一 Phase**。判据是"怎么验证做对了"，不是"做了什么"——没有可验证的证据就当这一步没完成。

**全局约定**：

- 文中所有具体数值（token 占用阈值、单工具输出上限、工具数量等）都是**起步默认值**，按场景实测调整，不是普适常数。
- 每个 Phase 标了正文对应章节（`↪`），需要展开原理时回正文对应位置 drill down。
- 接**强模型**做主力。弱模型带不动多轮工具调用，会在 Phase 1 的 gate 上反复失败。
- 这份 spec 走**折中覆盖档**：通用 runtime + 场景收窄 + 成本结构三段（来自实践沉淀）+ 把脉 + 四原则自检 + 机制准入闸门。Harness Lab 五层、可组合性三轴这两块进阶内容只给判定线、不展开，需要时回正文。

## Phase 0 · 把脉 · 动手前先摸模型脾气

**目标**：动手搭 runtime 之前，先给你要用的那个 LLM 端点跑一套诊断，产出一张画像——这个模型需要哪些机制、不需要哪些、哪些机制对它是陷阱。价值不在"了解模型"，在**砍掉后面要试的机制搜索空间** + **提前标出负贡献陷阱**。

**方法核心 · 探针三段式**：每条探针是一个三元组——

1. **刺激**：一个为暴露单一行为设计的极小任务（不求难，求能逼出这个行为）。
2. **行为分类**：判它"怎么做的"，不是判"对不对"。优先程序化判定（崩没崩、译没译、读了几次都是二元或可计数的），只在模糊维度才用模型当裁判。
3. **机制蕴含**：给定行为类，输出**一个配置决策 + 一个可被后面消融证伪的预测**。探针终结在配置决策 + 可验证预测，不终结在分数。

**四族探针**（按"出错后果"从硬到软分族，从 A 往 D 跑）：

- **A 族 · 协议层**：错了 harness 直接崩，不是退化。最高优先级，二元硬判定。典型——喂一个含嵌套对象 + 数组的复杂 strict schema 工具，看模型是在请求层直接失败，还是能正常注册调用。直接失败 → schema 归一化层必做。
- **B 族 · 工具使用层**：退化不崩。典型——给一个 tool-first 的多文件任务，数它先读几次再动手。反复读不动手 → 要上 read-complete / evidence-sufficiency guard。
- **C 族 · 指令遵循层**：典型——任务里用英文 heading，看模型在中文上下文里会不会把 heading 译成中文。会译 → verifier 不能只认英文固定字符串，要开多别名匹配。
- **D 族 · 自愈校准层**：最深、最差异化。典型——给一个会报错的操作，看模型是盲目重试、还是能读错误信息改方法。

**✅ 完成判据（gate）**：

- [ ] A/B/C/D 四族各跑了几条探针，每条都归到一个行为类（程序化判定的留下原始观察：崩了/译了/读了 N 次）。
- [ ] 每条探针产出了"配置决策 + 可证伪预测"两件，不是只留一个分数。
- [ ] 产出一张画像表：这个模型**必须开**哪些机制、**不需要**哪些、哪些是**陷阱**。这张表直接收窄 Phase 1-3 要做的机制集。

**注意**：把脉只给**定性先验**，不给定量结论。"开 X 能回收一部分被丢的工具调用"——这个"一部分"是多少，要等 Phase 3 消融拿干净数据回答。把脉看一眼就成立的是"它崩没崩 / 它译没译"这类事实，不依赖跨配置精确对照。

`↪ 正文：把脉 · Harness Lab 三步的第一步`

## Phase 1 · 通用 Agent Runtime · 不绑语言不绑场景

**目标**：搭一个能对着任意工具集发起 ReAct 循环的通用 runtime。这一步不碰具体业务，只把框架跑起来。

### 1.1 模块划分 · 五块

三大块加两块支撑：

- **engine**：跑循环。主循环就一个 ReAct——发消息 → 收回复 → 有工具调用就执行 → 结果回写 → 判终止。再加一个状态机管多轮：等用户输入 / 等用户确认 / 跑完 / 出错。每轮开头先查取消信号，stop 要即时生效。
- **llm**：做多供应商抽象。抽成一个统一接口，各家差异在适配层处理。`thinking` / reasoning 参数每家写法不同，用 `extra` 字段各自配。**国内 API 一律显式绕代理**，单独设 `no_proxy`——只设大写挡不住小写环境变量。
- **tools**：管注册和执行。每步重新加载工具列表**别缓存**——POST hook 里注册的新工具下一轮要看得见。
- **storage**：存 session 和 checkpoint。
- **types**：集中定义消息和工具签名。

`↪ 正文：Agent Loop（§5.1，P0）/ Model Adapter & Routing（§5.2，P0 边界）`

### 1.2 三条协议 / 运行时红线

- **assistant 消息协议**：`tool_calls` 后面必须**紧跟** `tool_result`，中间别插别的，否则下一轮请求直接 400。
- **strict schema 归一化**（如果 Phase 0 的 A 族探针显示模型对 strict schema 敏感）：注册工具前把 schema 规范化，含嵌套对象 / 数组的要处理到模型能接受。
- **取消信号**：每轮开头查，take 出 engine 后立即释放锁，异步跑并监听取消，成功 / 取消 / 出错三条路径都保证 engine 归还。别在锁里同步等长任务。

`↪ 正文：Model Adapter & Routing（§5.2）`

### 1.3 工具层

工具层不止"注册和执行"——**工具调用质量**（模型真的调、调对、结果干净回注、失败能自修）由下面几组共同决定。

**接口与防护**

- Tool 抽象按正文五字段定义（`name` / `description` / `input_schema` / `execute` / `policy`，policy 是 ToolPolicy：allowed_paths / timeout / requires_confirmation 等）。Registry 每次 tool_call 做三件事：schema 校验 → policy 决策（通过/人审/拒绝）→ 执行 + 审计进 trajectory。
- 单工具输出设**硬上限**（起步默认值：读文件 20k / 搜索 10k / shell 15k），超限走下面的「结果收口」。
- shell 和 git 命令一律走 `execFile` + 数组参数，**字符串拼接拒绝**，防注入。

**调度 · Tool Batch 四模式**（一批工具该并发、串行，还是停下）

- **parallel_read**：只读、无副作用、无路径冲突 → 默认并发（read_file / grep / glob / list_dir / web_search / web_fetch）。
- **sequential_write**：写文件 / 改工作区 / 变更状态 → 默认串行（write_file / edit_file / shell_exec / git_* 及任何标 DANGEROUS）。
- **barrier**：权限确认 / 危险操作 / 批次切换 / 需基于上一批观察重新决策 → 明确停下做切换点决策。
- **background_sidecar**：交轻量旁路 agent、与主线独立。
- **主线纪律**：单 agent 批量工具执行 → 结果收口回注 → 必要时才启 sidecar。别一上来就开 sub-agent（安全边界 / 上下文隔离 / 结果汇总三层成本，多数任务承担不起）。

**结果收口 · 工具结果不以原文直接进主对话**

- 批次执行后产出两类产物：raw 原文落 **ArtifactStore**（返 ref）+ **observation stub**（精简摘要，起步默认值约 200 字符，进 context）。主线程只消费 stub；要原文，模型自己再发 `read_observation(obs_id)` / `read_artifact` 按需取。口诀：**全文提取，按需注入，不截断，不跳页**。
- stub 必须带元信息 `truncated:true / size / preview_truncated_at`——避开两个对偶误区：**过载**（全塞 context，触发 lost-in-the-middle，花了 token 没拿到 attention）和**失真**（粗暴 truncate 丢信息、agent 不知被截）。**body 必须可寻址持久存储**，只 truncate 不存 body 仍是失真。
- 配套给 agent 的三句 prompt：① 反馈大时只看 stub，要完整内容用 `read_observation` 取；② stub 含 `truncated` 标记时看 `size` 判断要不要取全文；③ `PreprocessError` 类是 modal 处理失败、不是工具调用失败，可重试或换路径。

**错误返回 · 决定 agent 能不能自修**

- **actionable**：讲"哪里错了、怎么改"，不丢原始 stack trace。例：`file_path 'data/output.txt' 不存在，是不是 'data/input.txt'？先 list 一下 data/`——比抛一个原始 `FileNotFoundError` 强得多。它是把工具调对率从 60-70% 推到 95%+ 的 ACI 设计里关键一环（数字是整套 Tool Registry + ACI 合力，不是单靠这一条）。
- **raw vs sanitized 按工具来源切**：内部代码实现、execute 完全可控的工具 → raw error（含细节方便自修）；接外部数据源、execute 结果含外部内容的工具 → sanitized error（去 stack trace / 内部路径 / 敏感字段）。也可按环境切：开发 raw / 生产 sanitized。**错误返回是 prompt injection 最常见的注入点之一**。

**推荐默认配 web_search + web_fetch**

- 模型权重冻结、知识有 cutoff，cutoff 之后的事实（新版本号 / 刚改的 API / 当前文档）只能猜，猜不准就幻觉——这两个工具把"训练之后的世界"接进来现查现核，常比再调一轮 prompt 更能提质。
- 两条配套纪律：① 返回体量大（一次几十条 / 一篇几万字）→ 必须走上面的「结果收口」，别让原文灌进主对话；② 抓回内容是外部输入 → 过来源可信度判断或交 verifier，别把搜到的错误源直接当 ground truth。

**排查 · 工具该被调起却没调**

模型该调工具却只吐一段文本，三种成因落在三个层、对策完全不同——按响应解析 → 请求参数 → prompt 装配依次排除，比反复改 prompt 或换模型快得多：

1. **响应解析层（假阴性）**：模型可能把调用写成正文文本标签（如 `<tool_call>…</tool_call>`）而没进结构化 `tool_calls` 字段，Adapter 只认结构化字段就当普通文本丢了。对策：解析层在结构化字段之外再兜一道文本标签正则提取。
2. **请求参数层**：默认 `tool_choice:auto` 可调可不调。某回合必须走工具（必查库 / 必落盘）时把 `tool_choice` 设 `required` / `any` 或指定具体工具——**按回合按场景开，不全局常开**（长期 required 会逼模型在不该调时硬调制造噪声）。
3. **prompt 装配层**：过长的中文 prompt 会让某些模型"懒得"调工具直接用文本作答。把"要触发工具调用"的指令短而靠前、长说明拆分。

`↪ 正文：Tool Registry & ACI（§5.3，P0）/ ObservationPack · Observation Surface（§5.6）/ Model Adapter 三成因（§5.2 末小节）/ Artifact（§5.4，P2）`

### 1.4 上下文压缩 · 三种力度

- **micro**：压缩力度光谱最轻的一格——单个工具大结果的局部收口（stub 进 context / raw 落外置 / `read_observation` 按需取），具体机制见 1.3「结果收口」，这里只标它在三档压缩里的位置。
- **auto**：token 估算超窗口（起步默认值 60-70%，常见 70%）触发整段历史的小模型摘要。摘要**必须保留四类**——未闭合的 `tool_call_id` 及状态 / 每轮关键决策及理由 / artifact 引用 id / verifier 失败信息。丢任一类，agent 后续会幻觉编造（编假 observation、编假 id、再撞同一个错）。
- **rolling window**：最粗暴的兜底——超长直接滚掉最旧的轮次。
- **压完必须把任务目标和 system 身份重注入**，不然几十轮后 agent 会偏题，以为自己在干别的事。

`↪ 正文：Context（§5.4，P0）/ Prompt Assets 身份重注入（§5.5，P0）`

### 1.5 可观测性 · 从第一行就埋

- 埋在**决策点**不在执行点——机制判断"该不该做"的地方。
- 四种状态都记：触发执行了 / 条件不满足没执行 / 触发但拦截了 / 执行出错。
- **没事件是最危险的信号**——说明代码路径根本没走到。
- 循环检测拦重复工具调用，介入消息先攒到队列里，等 `tool_result` 处理完统一推，别中途破坏消息协议。

`↪ 正文：Observation Surface（§5.6）/ Trajectory（§5.7，P0）`

### 1.6 Safety 控制面起步 · 横切不是第 9 件

Safety 不是又一件 runtime，是横切在所有件之上的控制面——每次工具调用 / 状态变更 / 产物写入都被穿过，像 OS syscall gate，没有例外。完整是 **4 层权限决策模型**：

- **permission mode**：agent 整体运行模式（read-only / workspace-write / 等几档梯度）。
- **allow-deny-ask rules**：按工具按参数 pattern 的细粒度规则，走 **default deny + explicit allow**（允许 `git status` / 拒绝 `git push` / 询问 `git commit`）。
- **Hooks**：调用前后跑的用户脚本，做规则表达不了的复杂决策。走 **normalized intent 不走 raw 字符串**——否则 deny `cargo check` 会被 `cargo c` 之类 alias 绕过。
- **sandbox**：物理隔离层（Seatbelt / bubblewrap / container），限定文件读写范围 + network egress。前三层逻辑判断被绕过时它是最后防线。

起步至少先落两条：

- **关键 safety 走代码不走 LLM**（hard gate）：危险操作（删文件、推代码、发网络请求、花钱）的拦截判断写在代码层，不靠 prompt 让模型自觉。LLM 只在 soft gate（判 benign / harmful）上辅助，不做最后防线。
- **HITL approval gate**：high-impact 操作（删数据 / 转账 / 部署 / 发邮件）标 `requires_confirmation`，agent 调用前先发给人确认才执行，不让 agent 自己决定做不做。approval mode 沿父子 agent 链传递（sub-agent 默认继承 parent，不更宽松）。

`↪ 正文：Safety 控制面 · 4 层权限决策模型 + HITL（§5.9，cross-cutting）`

### 1.7 verifier 起步 · 先只上 hard gate

先上三层 verifier 里最便宜的一层——**hard gate**：程序化判定的 PASS/FAIL（pytest 通过 / build 成功 / 文件 hash 对）。outcome judge 和 PRM 等 Phase 2 再加。

`↪ 正文：Verifier 三层（§5.8，P0）`

### 1.8 前端接入（如果有）

后端是**唯一真相源**，别做前端状态快照。切换 session 从后端拉 `last_messages` 和 `is_running`。

**✅ Phase 1 完成判据（gate）**：

- [ ] runtime 能对着**任意工具集**发起 ReAct 循环，跑通至少一个 task。
- [ ] trajectory 有完整事件流，且事件埋在决策点、覆盖四状态。
- [ ] 危险命令（删 / 推 / 花钱）被**代码层**拦住——写一个会触发的 task 验证拦截真生效，不是 prompt 里写了"不要删"。
- [ ] high-impact 操作（删数据 / 转账 / 部署）触发 HITL，agent 调用前真的停下等人确认。
- [ ] hard gate 能对一个 task 判出 PASS/FAIL。
- [ ] 工具结果走收口——跑一个会返回大量内容（如 5000 行）的工具，确认主对话只进 stub、body 能被 `read_observation` 取回、stub 带 `size` / `truncated`。
- [ ] 连续跑几十轮不偏题（压缩后身份 + 目标重注入生效）。

## Phase 2 · 从通用到场景的收窄 · 不绑行业

**目标**：通用 runtime 跑通后，针对具体业务场景（客服 / 标书 / 质检 / 运维 / 翻译 / 数据分析等）收窄。方法对所有行业一样，八步做完。

1. **建评测集**：3-5 个典型任务起步，每个有可自动化的通过/失败判定、预期工具调用序列、最终数据库状态。评测集和代码一起演化，不要"做完再测"。
2. **工具收窄**：通用 runtime 可能挂几十个工具，业务场景一般只用一小撮（起步默认值 8-15 个）。其他**显式禁用**写白名单——留在上下文里会分散注意力、抬高选错概率，不是放着不调就行。
3. **工具描述强化**：每个业务工具的 description 直接列必填字段名、类型、枚举值。字段是 `user_id` 还是 `userId`、数组还是对象、枚举几个值，别让模型猜。**这是单点 ROI 最高的优化**，比改 system prompt 有效一个数量级。
4. **策略规则引擎**：业务规则不要全堆 system prompt（模型跑几十轮就忘）。做成"**调用 X 工具前注入结构化提醒**"——取消操作前弹资格检查、删除前弹二次确认。即时提醒比全局声明有效得多。
5. **提示词分层**：系统层写角色和能力边界，场景层写业务规则和禁用行为，执行层写输出格式。三层分开，方便单独调整和 A/B。
6. **开推理模式**（如果模型支持且 Phase 0 显示有收益）：多步判断 / 数值计算 / 规则检查密集的场景开，简单问答别开（贵、慢、无收益）。
7. **跑消融实验**：每加一个机制，关掉再跑一次评测集，看分数掉多少。正贡献保留，负贡献砍掉，接近零的看维护成本。关键在这步——有些"看起来正确"的机制关掉反而提升，单元测试看不出来。一个工具调用质量域的具体反例：**参数自动补全**（harness 替模型补缺省工具参数）看似帮模型省事，实测可能是负贡献——局部正确、全局有害，单点消融能抓出、单测看不见。消融要细到单机制 on/off，别整组开关。
8. **失败归因分维度**：不止看通过率，还要看——**该调的工具调没调**（没调 → 回 1.3「工具该被调起却没调」三处排查）/ 工具调用参数对不对 / 数据库最终状态对不对 / 回复里该说的数字/条款说了没 / 该走的流程走完没。单维度评测会掩盖真问题（数据库对了但没说出数字，在客户看来就是没办成事）。

### 2.1 什么时候从"收窄"升到"独立副 harness"

如果一个场景的领域知识塞进主 agent context 会撑爆，或者它要独立演化、独立迭代，就把它做成**独立副 harness**。判定标准是**5 维度本体齐不齐**——

1. **领域实体**：这个领域有哪些"东西"（如 PPT 副 harness 的 slide / layout / content_block）。
2. **实体属性**：每个实体有哪些字段 + 类型 + 约束。
3. **关系规则**：实体之间的逻辑约束（layout 决定 content_block 类型等）。
4. **状态机**：实体的合法状态 + 转换路径（draft → review → approved → exported）。
5. **操作集**：副 harness 暴露的工具集合。

**判定线**：5 维度齐 = 设计完成；少 1 维度算 80%；少 2 维度算 40%；少 3 维度 = 还在"万能 prompt"阶段，不算副 harness。只写一个 5000 字 system prompt 让模型自由发挥 ≠ 副 harness。

`↪ 正文：副 harness cell 5 维度本体（§8.5）`

### 2.2 verifier 从 hard gate 扩到三层

Phase 1 只上了 hard gate。这里按需补另两层：

- **第二层 · outcome judge**：用**另一个 LLM**对最终产出做语义评分。judge LLM 跟 agent LLM **跨厂商/跨 family**，防 preference leakage（同 family 互评会偏袒）。open-ended 任务才需要。
- **第三层 · PRM（过程评分）**：对推理过程做步骤级评分。长任务跑稳后再加，给消融提供 step-level 信号。
- **组合**：hard gate 不过，整体直接判 fail——不让模型在 fail 任务上靠 process 分混过。

`↪ 正文：Verifier 三层 + 三层组合策略（§5.8）`

**✅ Phase 2 完成判据（gate）**：

- [ ] 评测集每个任务都能**自动判定**通过/失败。
- [ ] 收窄后跑评测集，分数比通用版高，且**用消融证明**是收窄带来的（不是错觉）。
- [ ] 失败按多维度归因（参数 / 数据库状态 / 该说的话 / 流程），不是只看通过率。
- [ ] 若上了副 harness：5 维度本体齐全；若只是收窄：禁用工具写进白名单、关键规则做成调用前注入。
- [ ] 若上了 outcome judge：judge 跟 agent 跨 family。

## Phase 3 · 成本结构 + 机制准入 · 钱花在刀刃上

**目标**：资源不平摊。瓶颈在哪砸哪。顺序错了，花同样的钱拿不到同样的结果。

### 3.1 砸钱顺序

1. **先砸评测集**：没评测等于闭眼调优，后面所有改动都说不清"到底有没有用"。粗糙到只看通过率也比没有强。
2. **再砸工具描述**：单点 ROI 最高。模型猜字段名猜错一次浪费一轮，几十轮下来可能浪费一半轮数。
3. **接着砸策略规则引擎**：业务规则做成"调用前注入"，模型不会忘。这步做完再看表现，比直接换模型便宜得多。
4. **推理模式看任务**：多步判断 / 规则检查 / 数值计算密集的开，简单问答别开。
5. **模型升级放最后**：贵、不可逆（换了重跑评测）、对大部分场景提升有限。真正卡通过率的一般不是模型能力，是外围机制没调好。**第一步就换模型是最常见误区**——钱花了分数没涨。

**基础设施（压缩 / 观测 / 循环检测 / 锁和取消）是入场券不是投入**：不做一切都谈不上，但做到"能跑就行"够用，过度工程反而拖慢迭代。

### 3.2 按场景类型分配

| 场景类型 | 先砸哪里 |
|---|---|
| 客服 / 规则执行类 | 工具描述 + 策略规则，再考虑推理模式 |
| 代码 / 复杂规划类 | 推理模式 + 强模型，工具描述和规则次之 |
| 长对话 / 多轮类 | 压缩 + 观测，不然跑几十轮自己就崩 |
| 高并发 / 低延迟类 | 便宜模型 + 工具输出缓存 + 简化工具集 |

### 3.3 机制准入闸门 · 每个进默认 profile 的机制都要过

不能只凭"机制看起来合理"就默认启用。每个机制进主 harness 默认路径前，至少满足四条：

1. **能改变真实行为**：不是只增加 DTO、schema 或日志字段。
2. **有 on/off 消融**：关掉后能看到 pass_rate / cost / latency / reliability 的变化。
3. **有机器可读 trace**：记录触发原因、规则 ID、输入摘要、结果。
4. **过稳定性检查**：N 次重跑、扰动 prompt、打乱无关字段后，收益不能只靠偶然通过。

达不到四条的机制保留为**实验开关**，不进默认 profile。**负贡献机制要及时砍**——继续维护等于双重损失（占资源 + 拖通过率）。舍不得砍"看起来正确的代码"是常见错误，消融一测就看出来。

**✅ Phase 3 完成判据（gate）**：

- [ ] 砸钱按上面顺序走过，评测集在模型升级**之前**就建好了。
- [ ] 每个进默认 profile 的机制都过了准入四条（尤其有 on/off 消融数据）。
- [ ] 负贡献和接近零贡献的机制已经砍掉或降为实验开关。

`↪ 正文：起步建议四维度（各章末）/ 机制准入纪律（§七 Harness Lab）`

## 横切 · 四原则自检 · 每个 Phase 收尾跑一遍

每个 Phase 完成时，用控制论四原则各查一遍——任何一档失守都回去补，不要带着失守进下一 Phase。拿到任何 bug 也先归位到这四档之一。

- **可观测**：有没有 silent failure（错误被 try/catch 吞了不产生 event）？有没有 hidden state（agent 内部状态没对应 event）？`declared_vs_executed gap`（agent 声称做了 vs 工具/verifier 实际观测到的）持续偏高就是前哨信号。
- **可控**：控制点会不会被绕过（hook bypass）？工具粒度是不是过细让模型选不准？有没有"假落地"——机制在仓库里但生产路径是 noop？
- **稳定**：每件机制有没有 bound（max_turn / max_token / max_depth / max_retry）？这些 bound 都能监测吗？有没有 loop blind spot、context bloat、reward hacking？
- **闭环反馈**：每个改动有没有 ablation 数据支撑？"我觉得这样更好"但没数据 = 闭环失守。

## 进阶判定线 · 什么时候才上（不展开，需要时回正文）

这两块进阶能力默认**先不上**，满足判定线再上：

- **multi-agent / sub-agent fork-join**：先回答"single agent 跑这个任务 N=10 run 通过率多少？"没跑过 → 先跑 single agent；≥80% → 别上 multi-agent，优化 single agent 更值。multi-agent token 消耗约是普通 chat 的 15 倍（agent ~4× chat / multi-agent ~15× chat），coding 这类强依赖、可并行子任务少的场景尤其别上。
  - **若判定是必须上**（>60 turn、子任务真独立可验）：再问控制流可不可预知——哪些子步并行、谁交叉验证谁、结果怎么聚合，开跑前就清楚？**可预知** → 把这段编排写成**确定性脚本**交运行时后台跑（dynamic workflow），模型推理只发生在叶子 agent 干活时，lead 上下文最后只剩一个汇总答案，省掉 lead 即兴编排那块最烧的 token、且编排可重跑可审计；**不可预知**（探索性、下一步要看上一步结果）→ 只能让 lead 即兴编排，回到 15× 成本预期。gate：编排脚本能脱离某次具体 run 重跑、每步 `hands_off` / `calls_tool` 进 trajectory。
  - **边界**：dynamic workflow 降的是编排成本、不松准入门槛——single agent 优先、N=10 ≥80% 别上、coding 强耦合别上 这几条不变。
  `↪ 正文：Multi-Agent 过度分解 + dynamic workflow（§5.1.5 末段 + 脚注 / §5.1.6）/ fork-join 并发（§6.6）/ 拓扑轴（§八）`
- **dynamic harness · 副 harness 运行时路由**：§2.1"要不要造一个副 harness"之后的下一档——harness 跑起来后**按任务在多个副 harness 间动态选哪个**。先回答两件：① 你是不是真有 ≥2 个 5 维度本体齐全的副 harness（不齐 → 先把单个做齐，回 §2.1）；② 任务能不能靠"实体特征"而非"模糊文本"判该走哪个（判不出 → 还不到上路由的时候，单 harness 跑）。上了路由满足 3 条可验证纪律：routing 基于 task 实体特征 / 主 harness 只能 invoke 已挂载的副 harness 不能凭空造 / 每次 routing 决策写进 trajectory（写一个 task 验证 trajectory 里能查到"为什么走 A 不走 B"）。dynamic harness 跟上面的 dynamic workflow 是正交两件：前者管挂哪些工具/policy/verifier、后者管控制流怎么持有。**ReAct + dynamic harness + dynamic workflow** 这条组合是正文作者正在实践、仍在演进的方向，按当前实践给方向、不当定论。`↪ 正文：主↔副 routing 纪律（§8.7）/ dynamic harness（§5.1.6）`
- **Harness Lab 工作台（Observe-Score-Ablate-Tune-Iterate 五层）**：先回答"8 件 runtime + Safety 我项目里几件已经稳定？"≥6 件才考虑；harness 自己 pass rate 还跨周浮动 >10pp 时上 Ablate，跑出的信号 95% 是噪音。`↪ 正文：Harness Lab（§七）`
- **跨厂商协议（A2A 等）**：当前阶段建议自己定义 JSON-RPC schema + 文档化作内部标准，不要把架构绑在还在 evolve 的外部 spec 上。`↪ 正文：交互边界轴（§八）`

## 使用顺序

1. **Phase 0 把脉**先跑——便宜的定性诊断，把后面要试的机制搜索空间和陷阱先标出来。
2. **Phase 1** 搭通用 runtime，不考虑业务，把框架跑起来。gate 过了再走。
3. **Phase 2** 跟业务一起演化做收窄，不要等"通用版做到完美"再收窄。
4. **Phase 3** 任何阶段都能对照——它决定的不是"做什么"，是"做哪个先"和"花哪份钱"。
5. **四原则自检**每个 Phase 收尾各跑一遍。

总逻辑一句话：**先摸清模型脾气，再搭一个能跑的东西，针对场景削出锋利的刀，然后把钱花在刀刃上——每一步都用可验证的 gate 确认做对了，没有 gate 证据就当没做完。**
